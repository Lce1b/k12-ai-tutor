# 多模态K12人工智能通识课教学助手对话智能体

## 项目报告

---

### 一、作品概述

本作品面向"第二届浙江省大学生人工智能竞赛·揭榜挂帅赛"赛题JBGS-2026-02（多模态K12人工智能通识课教学助手对话智能体），由中国移动通信集团浙江有限公司发布。

针对K12阶段人工智能通识教育中优质师资供给不足、个性化教学资源匮乏、学生自主学习引导缺失三大核心痛点，我们构建了一套基于大语言模型的多模态对话智能体系统。系统以Orchestrator编排架构为核心，集成7个专业化Agent，支持4学段自适应、6种多模态交互、个性化学习路径，覆盖30节标准化AI课程，搭载226条分级知识的RAG检索引擎，为中小学生提供"7×24小时在线AI教师"服务。

**技术栈**：Next.js 14 + FastAPI + ChromaDB + OpenAI兼容LLM API

**代码规模**：后端~2500行（Python），前端~2000行（TypeScript），知识库226条。

#### 2.2 Agent编排策略

系统采用**Orchestrator + 专用Agent**的编排架构。Orchestrator负责意图分类和上下文管理，根据用户消息自动路由到对应Agent：

| Agent | 职责 | 触发条件 | 输出模态 |
|-------|------|----------|----------|
| Dialogue | 自由问答 | 默认路由 | Markdown文本 + RAG增强 |
| Teaching | 结构化授课 | "上课/学习/教我" | 微课卡片（引入→讲解→例子→互动→小结） |
| Animation | 概念动画化 | "动画/演示/可视化" | HTML5交互式动画 |
| Quiz | 练习测验 | "练习/测验/考考我" | 多题互动测验 + 即时批改 |
| Coding | 编程教学 | "代码/编程/写" | Python代码 + 沙箱执行 |
| PictureBook | 绘本生成 | "绘本/故事" | 绘本故事卡片 |
| Resources | 教学资源 | PPT/Word下载请求 | .pptx / .docx文件 |

**设计优势**：
1. **高内聚低耦合**：每个Agent独立负责一种交互模式，可单独优化和替换
2. **故障隔离**：单个Agent失败不影响其他功能，Orchestrator自动降级到Dialogue
3. **可扩展**：新增交互模式只需开发新Agent并注册到Orchestrator

#### 2.3 学段自适应机制

系统采用**三层自适应架构**，确保同一知识点对不同年龄段学生的呈现方式合理差异：

**第一层 — System Prompt注入**：
每个学段预设独立的system prompt，控制回复的语言风格、词汇难度、篇幅长度和内容深度。

**第二层 — 知识库分级**：
226条知识条目按4个学段分层标注，RAG检索时按学段过滤，确保检索到的内容匹配学生认知水平。

**第三层 — 参数化调整**：
根据学段动态调整LLM的max_tokens、temperature等参数。

四个学段的差异化设计：

| 维度 | 小学低(6-9岁) | 小学高(9-12岁) | 初中(12-15岁) | 高中(15-18岁) |
|------|--------------|---------------|--------------|--------------|
| 语言风格 | 故事比喻、温暖亲切 | 生活例子、鼓励动手 | 逻辑推理、连接学科 | 批判思维、学术严谨 |
| 篇幅 | ≤800字 | ≤1200字 | ≤1600字 | ≤2000字 |
| 数学深度 | 无公式 | 简单算术 | 基础代数 | 微积分/线代 |
| 代码要求 | 无 | 无 | Python入门 | 完整实现 |
| 交互方式 | 绘本/游戏为主 | 互动问答/简单实验 | Scratch/Python实践 | 项目式深度学习 |

### 三、大模型调用策略

#### 3.1 统一LLM接入层

系统采用OpenAI兼容API协议作为统一接入层，通过配置`LLM_BASE_URL`和`LLM_MODEL`可切换任意兼容服务（如OpenAI、DeepSeek、通义千问、智谱GLM、Moonshot等），无需修改代码。

#### 3.2 调用优化策略

**1. 意图分类用小模型**：Orchestrator的意图分类使用低temperature（0）、短max_tokens（100），确保快速准确。

**2. 结构化输出约束**：Teaching、Quiz、PictureBook、Resources等Agent统一使用`response_format={"type": "json_object"}`，将生成内容约束为结构化JSON，避免了解析歧义，提高了前端渲染的可靠性。

**3. RAG上下文注入**：Dialogue Agent在调用LLM前，先从ChromaDB检索3条最相关的知识条目，作为上下文注入prompt。这显著提高了回答的准确性和知识深度。

**4. 流式响应设计**：虽然当前版本采用同步REST，但架构已预留SSE流式接口的扩展空间，通过`StreamingResponse`可实现逐token输出，降低首字延迟。

**5. 容错降级**：所有Agent调用包裹在try-except中，LLM调用失败时自动降级到关键词匹配意图分类 + 通用对话回复，保证系统可用性。

#### 3.3 Token预算管理

| Agent | max_tokens | temperature | 说明 |
|-------|-----------|-------------|------|
| Orchestrator | 100 | 0 | 只需分类结果 |
| Dialogue | 1600 | 0.7 | 灵活但有控制 |
| Teaching | 2000 | 0.7 | 结构化JSON |
| Animation | 4000 | 0.8 | HTML代码较长 |
| Quiz | 2000 | 0.8 | 需要创造性出题 |
| Coding | 2000 | 0.7 | 代码示例 |
| PictureBook | 2000 | 0.9 | 需要故事创意 |
| Resources | 2000 | 0.7 | 结构化内容 |

### 四、多模态生成技术路线

#### 4.1 对话问答（①）

基于大语言模型 + RAG检索增强生成。用户提问经embedding后在ChromaDB向量库中检索相关知识条目，作为上下文注入LLM prompt，显著提升回答的专业性和准确性。

#### 4.2 多模态教学资源（②）

支持两种方式提供教学材料：
- **AI生成**：调用LLM生成结构化教学内容，通过python-pptx库构建PPT课件（6-8页），通过python-docx库构建Word文档（含学习目标、核心概念、详细讲解、生活例子、思考题）。作为兜底方案，同时实现了纯Python构建Office文档的零依赖方案（直接操作XML+ZIP），在缺少第三方库的环境中仍可运行。
- **后台上传**：支持PDF、DOC、PPT、TXT格式的教学资源上传。系统自动提取文本内容并索引到ChromaDB向量库，使上传资源可在对话中被RAG检索到。PDF提取使用PyPDF2，Office文档使用python-docx和python-pptx，纯文本直接读取。

#### 4.3 动画讲解（③）

通过prompt工程引导LLM生成纯HTML5/CSS/JS的教育动画（无需外部依赖）。动画类型包括：排序算法可视化（柱状图交换）、神经网络信号流动、梯度下降小球模拟、决策树分叉动画等。根据学段调整动画复杂度——低年级用可爱配色和比喻，高年级引入数学标注。

#### 4.4 绘本生成（④）

面向低龄学生，将AI知识点编排为图文并茂的绘本故事。LLM生成4页故事内容 + 每页的配图提示词（英文，适配Stable Diffusion/DALL-E等模型），可根据学段选择不同故事风格（童话/探险/科幻/科普）。

#### 4.5 编程环境（⑤）

提供三重编程支持：
- **代码生成**：根据学习主题生成教学代码示例（标注解释、预期输出、挑战任务）
- **预置模板**：5个可交互代码框架（冒泡排序、感知机AND逻辑、KNN分类器、梯度下降、决策树猜动物）
- **在线运行**：浏览器端提交代码→后端安全沙箱执行→返回输出结果。沙箱采用模块黑名单 + Python隔离模式（`-I`） + 超时10秒限制。

#### 4.6 游戏化练习（⑥）

多题互动测验系统，支持：题目逐题展示、即时批改反馈、正确/错误视觉差异化、正确率统计。在此基础上实现了游戏化机制：**XP经验值**（答对+10XP、答错+2XP）、**等级系统**（每100XP升1级）、**连击追踪**（连续答对计数，答错归零）、升级动画通知。

### 五、知识库构建方法

#### 5.1 知识体系设计

知识库围绕K12人工智能通识教育的核心主题，设计了6大模块、30节标准课程的教学大纲（curriculum.json），并据此构建了226条分级知识条目，覆盖：

| 模块 | 文件 | 条目 | 覆盖学段 |
|------|------|------|----------|
| AI基础概念 | ai_basics.json | 36 | 4学段全覆盖 |
| 数据与算法 | data_algorithms.json | 38 | 4学段全覆盖 |
| 机器学习入门 | machine_learning.json | 38 | 4学段全覆盖 |
| 计算机视觉 | computer_vision.json | 38 | 4学段全覆盖 |
| 自然语言处理 | nlp.json | 38 | 4学段全覆盖 |
| AI伦理与社会 | ai_ethics.json | 38 | 4学段全覆盖 |

#### 5.2 分级方法

每个知识点按4个学段分别撰写，遵循**认知发展梯度**：

- **小学低年级（6-9岁）**：故事化、角色化、比喻化。用"会思考的机器朋友"、"信息的小积木"等比喻降低理解门槛。200-300字。
- **小学高年级（9-12岁）**：生活实例驱动。用日常生活中的AI应用作为切入点，引入简单的逻辑和流程概念。250-350字。
- **初中（12-15岁）**：学科知识融合。连接数学、物理等学科知识，引入算法思想和基础代码。300-400字。
- **高中（15-18岁）**：学术化深度。引入数学推导、架构对比、前沿论文思想。350-500字。

#### 5.3 向量化与检索

使用Sentence-Transformers（all-MiniLM-L6-v2，中文优化版本）将知识条目编码为384维稠密向量，存入ChromaDB。检索时：用户query → embedding → cosine相似度计算 → 按学段过滤 → 返回Top-3结果。相似度使用cosine距离（有界[0,2]），避免了L2距离的无界问题。

### 六、个性化学习路径

#### 6.1 学习历史追踪

系统持久化记录每次学习交互，包括：会话消息数、学习的主题列表（去重+时间戳）、测验历史（每题的正确/错误+所属知识点）、答题正确率。

#### 6.2 掌握度自动判定

基于最近10次测验记录，自动计算每个知识点的正确率：
- 正确率 ≥ 80% → 标记为"已掌握"
- 正确率 ≤ 40% → 标记为"需加强"
- 需≥3次测验才进行判定，确保统计有效性

#### 6.3 兴趣偏好识别

追踪用户在不同知识点上的交互频率（消息数+测验次数），生成兴趣热度排名。Top兴趣主题用于个性化推荐——在欢迎语和学习建议中优先推荐用户感兴趣的方向。

#### 6.4 自适应学习闭环

```
学习 → 测验 → 掌握度分析 → 推荐下一步
  ↑                                    ↓
  └──────── 兴趣偏好更新 ←──────────────┘
```

- 薄弱知识点 → 推荐教学课程 + 动画演示 + 练习题
- 已掌握知识点 → 推荐进阶内容 + 编程实践
- 兴趣方向 → 推荐绘本/动画等趣味形式

#### 6.5 游戏化激励

- **XP经验系统**：答对+10XP，答错也+2XP（鼓励尝试）
- **等级机制**：每100XP升1级，等级越高所需XP越多
- **连击追踪**：连续答对计数，强化正反馈
- **经验条可视化**：侧边栏实时显示等级、XP进度条、连击数

### 七、遇到的难题及解决方案

#### 难题1：Office文档生成的依赖问题

**问题**：python-pptx和python-docx在某些Python环境中安装失败（尤其是Windows），导致PPT/Word生成功能不可用。

**解决方案**：实现了双轨方案——优先使用第三方库生成高质量文档，库不可用时自动降级为纯Python构建（直接操作Office Open XML格式的ZIP+XML结构）。验证了零依赖方案生成的.pptx和.docx文件在WPS和Microsoft Office中均可正常打开。

#### 难题2：代码沙箱的安全性

**问题**：在线Python代码执行存在安全风险——学生可能提交恶意代码（删除文件、网络攻击等）。

**解决方案**：三层防护：①模块黑名单（屏蔽os/subprocess/socket等危险模块，正则扫描代码）；②Python隔离模式（`python -I`忽略PYTHONPATH和用户site-packages）；③10秒超时强制终止。文档中明确标注此为"竞赛演示级"沙箱，生产环境建议使用Docker容器。

#### 难题3：中英文引号导致的JSON解析错误

**问题**：知识库条目中混入了ASCII双引号`"`用于中文引用语境（如"注意"），导致JSON解析失败。

**解决方案**：编写自动化修复脚本，通过正则`[一-鿿]"([一-鿿]+)"`匹配中文上下文中误用的ASCII引号，自动替换为中文直角引号「」。修复后在所有6个知识库文件上验证通过。

#### 难题4：前端与后端的类型不一致

**问题**：前端TypeScript中定义的API Response类型与后端实际返回的JSON结构不匹配（如学段标识符`elementary_low`vs`primary_low`、quiz结构从单题变为多题数组等），导致运行时渲染失败。

**解决方案**：逐字段对齐前后端接口定义，统一学段标识符为`primary_low/primary_high/middle/high`，统一quiz格式为`{questions: [...]}`数组结构，并在API封装层添加完整的TypeScript类型定义。构建阶段通过TypeScript编译检查确保类型安全。

#### 难题5：LLM输出格式不稳定

**问题**：部分LLM模型在`response_format={"type": "json_object"}`下仍偶尔输出非JSON内容（如包裹在```json代码块中）。

**解决方案**：在每个Agent中实现两层解析容错——先尝试直接`json.loads()`，失败后尝试正则提取代码块中的JSON，再失败则返回硬编码的兜底内容。确保前端在任何情况下都能获得可渲染的数据结构。

### 八、系统部署方案

本系统支持以下三种部署方式：

**方式一：本地开发部署**
```bash
# 后端
cd backend && pip install -r requirements.txt && uvicorn main:app --reload
# 前端
cd frontend && npm install && npm run dev
```
访问 http://localhost:3000

**方式二：生产部署**
```bash
# 后端（gunicorn + uvicorn worker）
cd backend && gunicorn -w 4 -k uvicorn.workers.UvicornWorker main:app
# 前端（静态导出）
cd frontend && npm run build && npm start
```

**方式三：Docker容器化部署**
```dockerfile
# 前端：next build → 静态文件 → nginx
# 后端：FastAPI → uvicorn
# 用docker-compose编排两个服务
```

系统采用前后端分离架构，前端通过Next.js的rewrites将`/api/*`代理到后端8000端口，避免跨域问题。LLM API Key通过环境变量注入，不写入代码或版本控制。

### 十、总结与展望

本作品构建了一个完整的多模态K12人工智能教学助手系统，核心创新在于**Agent编排架构**——将复杂的教学交互需求拆解为7个专职Agent，通过Orchestrator统一调度，既保证了各模态的专业性，又实现了系统的高内聚低耦合。226条分级知识库和30节课程大纲为教学内容提供了坚实的知识底座，游戏化积分系统和自适应学习路径提升了学习体验。

**未来改进方向**：接入语音对话（Whisper + TTS）、实现知识点关系图谱的可视化学习地图、基于学习数据的知识追踪模型（DKT/BKT）做更精准的认知诊断、支持多人协作学习场景。
